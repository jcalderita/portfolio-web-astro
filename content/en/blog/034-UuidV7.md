## My Problem 🤔

In my project I use <span class="high">PostgreSQL</span> as the database and <span class="high">UUID</span> as the identifier for all my models. It is the usual standard: universal, collision-free, secure. But I ran into a problem when my tables started to grow.

<span class="high">UUID v4</span> is completely random. This means that every new record is inserted at a random position within the <span class="high">B-Tree</span> index. The index becomes fragmented, pages split, and write performance degrades progressively. On tables with millions of records, the impact is real.

The solution has existed for a long time in databases like <span class="high">MongoDB</span> with <span class="high">ObjectID</span> or in systems that use <span class="high">ULID</span>: embed the timestamp in the identifier so that new records are always inserted at the end of the index.

<span class="high">UUID v7</span>, defined in <span class="high">RFC 9562</span>, brings exactly that to the UUID standard. <span class="high">PostgreSQL 18</span> already includes the native <span class="high">uuidv7()</span> function. But Swift — and by extension <span class="high">Fluent</span> — still generates <span class="high">UUID v4</span> by default with <span class="high">UUID()</span>.

If the identifier is generated by my application before persisting it, I am losing the benefits of v7 even though my database already supports it.

---

## My Solution 🧩

I decided to create an extension on <span class="high">UUID</span> that implements <span class="high">UUID v7</span> generation directly in Swift, following the <span class="high">RFC 9562</span> specification.

### Monotonic state

The first thing I need is a shared state that guarantees two UUIDs generated in the same millisecond are distinct and ordered. I use a <span class="high">Mutex</span> from the <span class="high">Synchronization</span> framework to make it thread-safe without resorting to <span class="high">DispatchQueue</span> or <span class="high">actor</span>:

```swift
import Foundation
import Synchronization

private let v7State = Mutex(V7State())

private struct V7State: Sendable {
    var lastTimestamp: UInt64 = 0
    var sequence: UInt16 = 0
}
```

### UUID generation

The main extension implements the static method <span class="high">v7()</span> that generates a UUID conforming to <span class="high">RFC 9562</span>:

```swift
extension UUID {
    public static func v7() -> UUID {
        var bytes = (
            UInt8(0), UInt8(0), UInt8(0), UInt8(0),
            UInt8(0), UInt8(0), UInt8(0), UInt8(0),
            UInt8(0), UInt8(0), UInt8(0), UInt8(0),
            UInt8(0), UInt8(0), UInt8(0), UInt8(0)
        )

        let (timestamp, seq) = v7State.withLock { state -> (UInt64, UInt16) in
            var now = UInt64(Date().timeIntervalSince1970 * 1000)

            if now == state.lastTimestamp {
                if state.sequence >= 0x0FFF {
                    while now == state.lastTimestamp {
                        now = UInt64(Date().timeIntervalSince1970 * 1000)
                    }
                    state.lastTimestamp = now
                    state.sequence = UInt16.random(in: 0...0x0FFF)
                } else {
                    state.sequence += 1
                }
            } else {
                state.lastTimestamp = now
                state.sequence = UInt16.random(in: 0...0x0FFF)
            }

            return (now, state.sequence)
        }

        // Bytes 0–5: 48-bit timestamp (big-endian)
        bytes.0 = UInt8(truncatingIfNeeded: timestamp >> 40)
        bytes.1 = UInt8(truncatingIfNeeded: timestamp >> 32)
        bytes.2 = UInt8(truncatingIfNeeded: timestamp >> 24)
        bytes.3 = UInt8(truncatingIfNeeded: timestamp >> 16)
        bytes.4 = UInt8(truncatingIfNeeded: timestamp >> 8)
        bytes.5 = UInt8(truncatingIfNeeded: timestamp)

        // Byte 6: version (0x7_) | top 4 bits of sequence
        bytes.6 = 0x70 | UInt8(seq >> 8)

        // Byte 7: lower 8 bits of sequence
        bytes.7 = UInt8(truncatingIfNeeded: seq)

        // Bytes 8–15: random, then set variant
        var random: UInt64 = 0
        withUnsafeMutableBytes(of: &random) { buf in
            _ = SecRandomCopyBytes(kSecRandomDefault, buf.count, buf.baseAddress!)
        }
        bytes.8  = UInt8(truncatingIfNeeded: random >> 56)
        bytes.9  = UInt8(truncatingIfNeeded: random >> 48)
        bytes.10 = UInt8(truncatingIfNeeded: random >> 40)
        bytes.11 = UInt8(truncatingIfNeeded: random >> 32)
        bytes.12 = UInt8(truncatingIfNeeded: random >> 24)
        bytes.13 = UInt8(truncatingIfNeeded: random >> 16)
        bytes.14 = UInt8(truncatingIfNeeded: random >> 8)
        bytes.15 = UInt8(truncatingIfNeeded: random)

        // Byte 8: variant 0b10xx_xxxx
        bytes.8 = (bytes.8 & 0x3F) | 0x80

        return UUID(uuid: bytes)
    }
}
```

The structure of the generated UUID follows the <span class="high">RFC 9562</span> specification:

- **Bytes 0–5** (48 bits): <span class="high">timestamp</span> in milliseconds since epoch, in big-endian. This is what guarantees chronological order.
- **Byte 6** (high nibble): version <span class="high">0x7</span> — identifies the UUID as v7.
- **Bytes 6–7** (lower 12 bits): sequence counter. Guarantees monotonicity within the same millisecond.
- **Bytes 8–15** (64 bits): random bytes generated with <span class="high">SecRandomCopyBytes</span>, with the two upper bits of byte 8 forced to <span class="high">0b10</span> to mark the <span class="high">RFC 4122</span> variant.

The monotonic logic inside <span class="high">withLock</span> works as follows: when two calls occur within the same millisecond, the counter is incremented. If the counter reaches its maximum (<span class="high">0x0FFF</span>, i.e., 4095 UUIDs in a single millisecond), the implementation actively waits for the next millisecond before continuing — it never generates two identical UUIDs.

### Inspection utilities

In addition to generation, I added two properties to inspect any UUID: one that checks whether it is v7 and another that extracts the embedded <span class="high">timestamp</span>:

```swift
extension UUID {
    public var isV7: Bool {
        (uuid.6 >> 4) == 0x07 && (uuid.8 >> 6) == 0x02
    }

    public var v7Timestamp: Date? {
        guard isV7 else { return nil }

        let ms = UInt64(uuid.0) << 40
            | UInt64(uuid.1) << 32
            | UInt64(uuid.2) << 24
            | UInt64(uuid.3) << 16
            | UInt64(uuid.4) << 8
            | UInt64(uuid.5)

        return Date(timeIntervalSince1970: Double(ms) / 1000.0)
    }
}
```

<span class="high">isV7</span> checks that the high nibble of byte 6 is <span class="high">0x07</span> (version 7) and that the two upper bits of byte 8 are <span class="high">0b10</span> (RFC 4122 variant). <span class="high">v7Timestamp</span> reconstructs the 48-bit timestamp from bytes 0–5 and converts it into a <span class="high">Date</span>.

## My Result 🎯

Adoption is immediate. In any <span class="high">Fluent</span> model, I simply replace <span class="high">UUID()</span> with <span class="high">UUID.v7()</span> in the initializer:

```swift
// Before
self.id = UUID()

// After
self.id = UUID.v7()
```

The database type does not change — it is still <span class="high">UUID</span>. <span class="high">Fluent</span> requires no modifications. The change is completely transparent to the ORM.

The benefits I have gained are straightforward:

- **Ordered indexes** — new records are always inserted at the end of the <span class="high">B-Tree</span>, eliminating fragmentation
- **Native chronological sorting** — <span class="high">ORDER BY id</span> is equivalent to <span class="high">ORDER BY created_at</span> without an extra column
- **Embedded timestamp** — I can extract when a record was created directly from the UUID with <span class="high">v7Timestamp</span>
- **Backward compatible** — the format remains standard UUID; any system that accepts <span class="high">UUID v4</span> accepts <span class="high">v7</span>
- **Thread-safe** — it works correctly from multiple concurrent threads thanks to the <span class="high">Mutex</span>

```swift
let id = UUID.v7()

print(id.isV7)          // true
print(id.v7Timestamp!)  // the creation date embedded in the UUID
```

And if you are already on <span class="high">PostgreSQL 18</span>, the native <span class="high">uuidv7()</span> function generates UUIDs with the same structure. UUIDs generated from Swift and from PostgreSQL are interchangeable and comparable with each other.

The change from <span class="high">UUID()</span> to <span class="high">UUID.v7()</span> is a one-liner. The benefits, permanent.
